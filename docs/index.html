01 Arquitectura general del sistema

El sistema sigue un modelo cliente-servidor con tres componentes principales. Cada agente ejecuta una extensión de Chrome que se comunica vía WebSocket con un servidor central Node.js. Un dashboard web administrativo consume la misma información a través de una API REST.

Arquitectura general del sistema: la extensión se comunica en tiempo real con el servidor WebSocket; el dashboard consulta datos históricos vía REST; SQLite persiste toda la información.

🟣 Extensión de Chrome

Se inyecta dentro de WhatsApp Web. Detecta el contacto abierto, muestra un panel flotante con el estado de todos los agentes, y envía actualizaciones al servidor.

🟢 Servidor WebSocket + REST

Centraliza las conexiones de todos los agentes, autentica por PIN, guarda sesiones en SQLite, y retransmite los cambios de estado a todos los clientes conectados.

🔵 Dashboard Administrativo

Panel web visible en http://localhost:3000. Muestra en vivo el estado de cada agente, el historial de sesiones, y reportes gráficos de productividad.

02 Flujo de comunicación

Cuando un agente abre la extensión y se conecta, se establece un canal WebSocket bidireccional. Cada cambio de estado o detección de contacto se transmite al servidor, que lo reenvía a todos los demás agentes.

Diagrama de secuencia — conexión y atención de un chat
sequenceDiagram
    participant E as Extensión
    participant SR as Servidor WebSocket
    participant DB as SQLite
    participant O as Otros Agentes

    Note over E,SR: 1. Autenticación
    E->>+SR: ATTENDING { agent: "Ana", pin: "1234" }
    SR->>DB: Verificar PIN en agents
    DB-->>SR: PIN válido
    SR-->>-E: WELCOME + SERVER_INFO
    SR->>O: PRESENCE_UPDATE (Ana → available)

    Note over E,SR: 2. Agente abre un chat
    E->>+SR: ATTENDING { agent: "Ana", contact: "Cliente X" }
    SR->>DB: INSERT session (start_time, contact)
    SR-->>-O: PRESENCE_UPDATE (Ana → active, contact: "Cliente X")

    Note over E,SR: 3. Heartbeat (cada 30s)
    E->>SR: HEARTBEAT { agent: "Ana" }
    SR->>DB: UPDATE last_seen
    SR-->>E: (no broadcast)

    Note over E,SR: 4. Agente cierra chat
    E->>+SR: AVAILABLE { agent: "Ana" }
    SR->>DB: UPDATE session (end_time, duration)
    SR-->>-O: PRESENCE_UPDATE (Ana → available)
Secuencia completa desde que el agente se autentica, atiende un chat, y vuelve a disponible. El servidor persiste cada evento en SQLite y notifica a los demás agentes en tiempo real.

Protocolo de mensajes WebSocket

DirecciónTipoPropósito
Cliente → ServidorATTENDINGAgente comenzó a atender un chat
Cliente → ServidorAVAILABLEAgente está libre/disponible
Cliente → ServidorPAUSEDAgente pausó su disponibilidad
Cliente → ServidorHEARTBEATKeepalive (cada 30 segundos)
Cliente → ServidorDELETE_AGENTEliminar agente del servidor
Servidor → ClienteWELCOMEBienvenida al conectarse
Servidor → ClienteSERVER_INFOEstado inicial completo del sistema
Servidor → ClientePRESENCE_UPDATECambio de estado de algún agente
Servidor → ClienteERRORError (autenticación, formato, etc.)
Cada mensaje es un JSON con tipo y payload. El servidor valida, procesa, persiste y retransmite según corresponda.

03 Base de datos — esquema relacional

SQLite almacena toda la información de manera persistente. Cuatro tablas principales normalizadas que permiten consultar historial de sesiones, actividad por agente, y métricas de productividad.

Diagrama entidad-relación
erDiagram
    agents {
        TEXT id PK "UUID único"
        TEXT name "Nombre del agente"
        TEXT pin_hash "SHA-256 del PIN"
        TEXT color "Color para el avatar"
        INTEGER created_at "Timestamp de registro"
        INTEGER active "1=activo, 0=eliminado"
    }

    sessions {
        INTEGER id PK "Autoincrement"
        TEXT agent_id FK "→ agents.id"
        INTEGER start_time "Inicio de la atención"
        INTEGER end_time "Fin de la atención (NULL si activa)"
        TEXT contact "Nombre del contacto atendido"
        TEXT status "active | available | paused"
        INTEGER duration "Diferencia end - start (segundos)"
    }

    activity_log {
        INTEGER id PK "Autoincrement"
        TEXT agent_id FK "→ agents.id"
        INTEGER timestamp "Momento del evento"
        TEXT event_type "ATTENDING | AVAILABLE | PAUSED | OFFLINE"
        TEXT details "JSON con información adicional"
    }

    auth_pins {
        INTEGER id PK "Autoincrement"
        TEXT agent_id FK "→ agents.id"
        TEXT pin_hash "SHA-256 del PIN"
        INTEGER created_at "Fecha de creación"
        INTEGER expires_at "Fecha de expiración (NULL = vigente)"
    }

    agents ||--o{ sessions : tiene
    agents ||--o{ activity_log : genera
    agents ||--o{ auth_pins : posee
Cuatro tablas: agents almacena los perfiles, sessions registra cada atención, activity_log guarda un historial detallado de eventos, y auth_pins gestiona los PINs de autenticación.

📋 agents

Registro maestro de agentes. Cada agente tiene un ID único, nombre, hash de PIN, y un indicador de activo/inactivo.

📞 sessions

Cada atención a un chat es una sesión. Guarda inicio, fin, contacto, duración. Permite calcular tiempo promedio de atención.

📝 activity_log

Bitácora cronológica de cada cambio de estado. Usado para reconstruir la línea del tiempo de cada agente.

🔑 auth_pins

Gestiona los PINs de acceso con hash SHA-256. Soporta expiración y múltiples PINs por agente (rotación).

04 Autenticación por PIN

Cada agente tiene un PIN único de 4-6 dígitos. Al conectarse, la extensión envía el PIN junto con el nombre del agente. El servidor valida contra el hash almacenado en SQLite antes de permitir el acceso.

Diagrama de secuencia — autenticación
sequenceDiagram
    participant Popup as Popup Extensión
    participant SW as Service Worker
    participant SR as Servidor WS
    participant DB as SQLite

    Popup->>Popup: Usuario selecciona agente\n e ingresa PIN
    Popup->>SW: POPUP_READY { agentName, pin }
    SW->>SW: Abre WebSocket
    SW->>SR: ATTENDING { agent: "Ana", pin: "1234" }
    SR->>DB: SELECT pin_hash FROM auth_pins\nJOIN agents WHERE name = "Ana"
    DB-->>SR: pin_hash encontrado
    SR->>SR: SHA-256("1234") == pin_hash?
    alt PIN correcto
        SR-->>SW: WELCOME + SERVER_INFO
        SW-->>Popup: CONNECTION_STATUS { connected: true }
        Popup->>Popup: Mostrar "Conectado"
    else PIN incorrecto
        SR-->>SW: ERROR { code: "AUTH_FAILED" }
        SW-->>Popup: Mostrar error
        Popup->>Popup: Solicitar PIN nuevamente
    end
El PIN nunca viaja en texto plano después del handshake inicial. El servidor lo valida contra el hash almacenado. Tres intentos fallidos bloquean temporalmente al agente.

05 Ciclo de vida del agente — máquina de estados

Cada agente solo puede estar en un estado a la vez. Las transiciones están controladas tanto por acciones del usuario (abrir/cerrar chat, pausar) como por el sistema (timeout de heartbeat).

Diagrama de estados del agente
stateDiagram-v2
    [*] --> Desconectado
    Desconectado --> Disponible: Conectar (PIN válido)

    Disponible --> Atendiendo: Abre chat\n(ATTENDING)
    Disponible --> Pausado: Pausa manual\n(PAUSED)
    Disponible --> Desconectado: Cierra navegador\n(OFFLINE)

    Atendiendo --> Disponible: Cierra chat\n(AVAILABLE)
    Atendiendo --> Pausado: Pausa manual\n(PAUSED)
    Atendiendo --> Desconectado: Pérdida de conexión\n(heartbeat timeout)

    Pausado --> Disponible: Reanuda\n(AVAILABLE)
    Pausado --> Desconectado: Cierra navegador\n(OFFLINE)

    Desconectado --> [*]: Eliminar agente\n(DELETE_AGENT)

    note right of Atendiendo
        El timer de atención
        se inicia al entrar
        a este estado
    end note

    note right of Desconectado
        Heartbeat 30s
        Timeout 60s → offline
    end note
El agente inicia Desconectado, pasa a Disponible al autenticarse, y transiciona entre Atendiendo y Pausado según su actividad. La pérdida de conexión o timeout lo regresa a Desconectado.

Métricas del módulo de reportes

MétricaCálculoVisualización
Tiempo promedio de atenciónAVG(end_time - start_time) de sesiones cerradasGráfica de barras por agente
Agentes más activosCOUNT(sessions) GROUP BY agent_idTop 5 ranking con medallas
Horas picoGROUP BY strftime('%H', start_time)Gráfica de líneas (distribución horaria)

Todas las métricas se calculan con consultas SQL sobre la tabla sessions y se grafican con Chart.js.

Comparativa: antes vs. después

De no existir el sistema, un equipo de agentes en WhatsApp Web opera sin ninguna coordinación ni visibilidad.

AspectoAntes (sin el sistema)Después (con el sistema)
Estado de los agentes ✗ Desconocido — los agentes preguntan "¿estás libre?" por otro canal ✓ Visible en tiempo real: quién está atendiendo, disponible, pausado o desconectado
Coordinación de chats ✗ Dos agentes pueden atender al mismo cliente sin saberlo ✓ Cada agente ve qué contacto está siendo atendido y por quién
Historial de actividad ✗ No existe — no hay registro de qué se atendió ✓ Cada sesión queda registrada en SQLite con fechas, contactos y duración
Control de acceso ✗ Cualquiera puede decir que es "Agente 1" ✓ PIN único por agente validado contra hash en base de datos
Métricas de productividad ✗ No hay forma de medir rendimiento del equipo ✓ Reportes automáticos: tiempo promedio, horas pico, top agentes
Persistencia de datos ✗ Al reiniciar el servidor se pierde todo ✓ SQLite guarda todo — al reiniciar los datos siguen ahí
Dashboard administrativo ✗ No existe ✓ Panel web con 4 vistas: en vivo, historial, reportes, gestión
Calidad del software ✗ Sin pruebas automatizadas ✓ Suite completa de tests con Vitest (unit + integration)
Despliegue ✗ Solo local, requiere instalar dependencias manualmente ✓ Docker Compose — un solo comando levanta todo
El sistema transforma un entorno de trabajo sin visibilidad en una plataforma coordinada, medible y profesional.
