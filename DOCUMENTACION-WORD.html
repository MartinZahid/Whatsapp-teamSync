1. Visión General del Sistema

1.1 ¿Qué problema resuelve?

En equipos de servicio al cliente que usan WhatsApp Web, varios agentes atienden chats simultáneamente. Sin un sistema de coordinación, es común que:

  • ✅ Dos agentes atiendan al mismo cliente sin saberlo
  • ❌ Un cliente quede sin atender porque nadie sabe quién está libre
  • ❌ No haya visibilidad de quién está trabajando y quién está en pausa

WhatsApp Team Sync resuelve esto mostrando un panel flotante sobre WhatsApp Web con el estado en tiempo real de cada agente del equipo.

✅ Valor Principal Coordinación visual en tiempo real para equipos de soporte sin necesidad de herramientas externas.

SERVIDOR WEB SOCKET — FLUJO PRINCIPAL ┌───────────────────────────────────────────────────────────────────┐ │ │ │ Inicio │ │ │ │ │ ▼ │ │ Crear WebSocketServer en puerto 3001 │ │ │ │ │ ├──► Iniciar intervalo de heartbeat (cada 30s) │ │ │ │ │ │ │ ▼ │ │ │ roomManager.checkHeartbeats() │ │ │ │ │ │ │ └──► Si agente sin actividad > 60s → marcar offline │ │ │ │ │ └──► Esperar conexiones entrantes │ │ │ │ │ ▼ │ │ Conexión entrante │ │ │ │ │ ▼ │ │ Enviar WELCOME {"type":"WELCOME", ...} │ │ │ │ │ ▼ │ │ Esperar primer mensaje (autenticación) │ │ │ │ │ ├──► ¿Es ATTENDING con agent name? │ │ │ │ │ │ │ ├──► Sí → roomManager.register(ws, name) │ │ │ │ │ │ │ │ │ ▼ │ │ │ │ isAuthenticated = true │ │ │ │ roomManager.sendInitialState(agentId) │ │ │ │ │ │ │ └──► Modo conversación: │ │ │ │ │ │ │ ▼ │ │ │ Mensaje del agente: │ │ │ │ │ │ │ ├──► ATTENDING → status='active' │ │ │ ├──► PAUSED → status='paused' │ │ │ ├──► AVAILABLE → status='available' │ │ │ ├──► OFFLINE → status='offline' │ │ │ └──► DELETE_AGENT → eliminar agente │ │ │ │ │ │ │ ▼ │ │ │ roomManager.broadcastPresence() │ │ │ │ │ └──► No → Enviar ERROR AUTH_REQUIRED │ │ │ │ Cierre de conexión: │ │ │ │ │ ▼ │ │ roomManager.disconnect(agentId) │ │ │ │ │ └──► Si no hay remplazo → marcar offline │ │ │ │ Graceful Shutdown (SIGINT/SIGTERM): │ │ wss.close(() → process.exit(0)) │ └───────────────────────────────────────────────────────────────────┘

Ciclo de vida de una conexión

PasoEventoAcción del Servidor
1connectionVariables: agentId = null, isAuthenticated = false. Envía WELCOME. Configura ping cada 30s.
2message (1ro)Debe ser ATTENDING con agent. Si válido → register() + autenticado. Si no → ERROR.
3message (siguientes)roomManager.handleMessage(agentId, message) según tipo.
4closeroomManager.disconnect(agentId) + limpiar ping.
5errorSolo loguea el error.

4.2 Service Worker (background)

Archivo: extension/src/background/service-worker.ts

Clase: BackgroundManager

El Service Worker (SW) es el cerebro de la extensión. Vive de forma persistente en segundo plano y gestiona:

  • 🔌 La conexión WebSocket con el servidor
  • 🔄 Reconexión automática con backoff exponencial
  • 💓 Heartbeat cada 30s para mantener la conexión viva
  • 📡 Enrutamiento de mensajes entre popup, content scripts y servidor
  • 💾 Persistencia de configuración en chrome.storage.local

Variables de instancia

configAgentConfig | nullConfiguración: agentName + serverUrl
agentsMap<string, AgentState>Estado local de todos los agentes
currentTabIdnumber | nullID de la pestaña activa actual
reconnectAttemptsnumberContador de intentos de reconexión (máx 10)
isIntentionallyClosedbooleanSi el cierre fue intencional (no reconectar)
heartbeatTimernumber | nullID del interval de heartbeat

4.3.3 DomObserver

Archivo: extension/src/content/dom-observer.ts

Clase: DomObserver

Observa cambios en el DOM de WhatsApp Web para detectar cuándo un agente selecciona o deselecciona un chat.

Selectores CSS utilizados

ConstanteSelectorPropósito
CHAT_LIST_SELECTORdiv[data-testid="chat-list"]Contenedor de la lista de chats

Métodos principales

MétodoDescripción
onChatSelect(callback)Registra callback para cuando se selecciona un chat. Devuelve función para desuscribirse.
onChatDeselect(callback)Registra callback para cuando se deselecciona un chat.
restart()Reinicia el observer (útil cuando WhatsApp recarga la SPA).
stopObserving()Detiene la observación y limpia recursos.
isActive()Devuelve si el observer está activo.

Estrategia de detección

El DomObserver usa 3 mecanismos simultáneos:

  1. MutationObserver — Detecta cambios de atributos (aria-selected, tabindex) y nuevos elementos en la lista
  2. Click listeners — Detects clicks directos en los items del chat
  3. Debounce — 100ms para selección, 300ms para deselección (evita falsos positivos)

Además, observa cambios en la URL (las SPAs cambian la URL sin recargar la página) y reinicia el observer cuando detecta una navegación.

5. Protocolo de Comunicación

Toda la comunicación usa JSON sobre WebSocket. Cada mensaje tiene un campo type que identifica su propósito.

5.1 Mensajes Cliente → Servidor

TipoEjemplo¿Cuándo se envía?
ATTENDING{"type":"ATTENDING","agent":"Pedro","contact":"Doña Rosa","status":"active"}Cuando un agente abre un chat
PAUSED{"type":"PAUSED","agent":"Pedro","reason":"user_paused"}Cuando un agente se pausa
AVAILABLE{"type":"AVAILABLE","agent":"Pedro"}Cuando un agente vuelve a estar disponible
OFFLINE{"type":"OFFLINE","agent":"Pedro"}Cuando un agente se desconecta
DELETE_AGENT{"type":"DELETE_AGENT","agent":"Pedro"}Cuando se elimina un agente permanentemente
HEARTBEAT{"type":"HEARTBEAT","agent":"Pedro"}Cada 30s para mantener la conexión viva

TipoEjemplo¿Cuándo se envía?

PRESENCE_UPDATE{"type":"PRESENCE_UPDATE","agents":[...]}Cada vez que cambia el estado de algún agente
SERVER_INFO{"type":"SERVER_INFO","version":"1.0.0","agents":[...]}Cuando un agente se autentica (estado inicial)
ERROR{"type":"ERROR","code":"AUTH_REQUIRED","message":"..."}Cuando hay un error (autenticación, parseo, etc.)
